Git 工作流的艺术

从 Conventional Commits 到分支策略 — 适合个人开发者的高效 Git 实践方案

2026.06.18 · 作者分享

从 Conventional Commits 到分支策略，一套适合个人开发者的高效 Git 实践方案。

一、Conventional Commits — 提交信息规范化

规范化提交信息不仅让 Git 历史清晰可读，还能自动生成 Changelog 和版本号。Conventional Commits 是目前最广泛采用的规范。

feat: 添加用户登录功能
fix: 修复导航栏在移动端的显示问题
docs: 更新 README 安装说明
style: 统一代码缩进格式
refactor: 重构用户验证逻辑
test: 添加登录模块单元测试
chore: 更新依赖版本
💡 核心格式

type(scope): description — type 表示变更类型，scope 是可选的模块范围，description 为简洁描述。

二、分支策略

对于个人项目，推荐采用简化版的 Trunk-Based Development：主分支保持稳定可部署，功能开发在特性分支上进行，完成后合并回主分支。

01
主分支 master / main

始终保持可部署状态。不直接在 master 上提交代码。

02
特性分支 feat/xxx

每个功能或修复在独立分支开发，完成后通过 merge 或 rebase 合入主分支。

03
热修复 hotfix/xxx

紧急修复从 master 切出，修复后快速合回并打 tag。

三、实用技巧

交互式 Rebase 整理提交历史

git rebase -i HEAD~3  # 整理最近 3 次提交

在推送前用 rebase 将多个零散的 WIP 提交合并为一个有意义的提交，保持历史整洁。

别名提高效率

git config --global alias.lg "log --oneline --graph --all"
git config --global alias.st "status -s"
git config --global alias.co "checkout"

四、总结

好的 Git 工作流不在于规则多么复杂，而在于一致性和可追溯性。坚持 Conventional Commits 规范 + 简洁的分支策略，足以让个人项目的版本管理井井有条。
